SQLite и быстрый CRUD GUI для Актов работ

Документ фиксирует, как адаптировать архитектуру актов под SQLite и какой GUI выбрать для максимально быстрого CRUD без разрушения истории документов.

1. Короткий вывод

База: SQLite подходит для старта, одиночного инстанса, небольшого числа одновременных записей и локального/внутреннего сервиса.

CRUD GUI: лучший быстрый вариант — NocoDB.

Важно: NocoDB не должен напрямую ломать immutable-историю; операции freeze revision, sign, send, render лучше проводить через backend/API.

2. Когда SQLite подходит

  • Один сервер или десктопное/локальное приложение.
  • Небольшое количество конкурентных write-операций.
  • История документов важнее горизонтального масштабирования.
  • Нужен быстрый старт без отдельного DB-сервера.

Для учёта актов это нормальный выбор, если система не предполагает много параллельных операторов, массовые фоновые записи и распределённую кластерную запись.

Для денег в SQLite лучше всего использовать целые minor units: например, 105055 = 1050.55 ₽. Это надёжнее, чем надеяться на десятичную точность SQLite.

5. Что меняется в нашей схеме

  • uuid → TEXT
  • timestamptz → TEXT в UTC
  • jsonb → TEXT + json_valid()
  • decimal(18,2) для сумм → INTEGER в копейках
  • Partial index для текущей ревизии остаётся допустимым, если версия SQLite современная

6. Ограничения и caveats SQLite

  • SQLite — это single-writer база; длинные write-транзакции вредны.
  • Нельзя держать транзакцию открытой во время рендера HTML или сетевого вызова ЭДО.
  • ALTER TABLE ограничен сильнее, чем в PostgreSQL; часто нужен rebuild таблицы.
  • При росте нагрузки по записи может понадобиться миграция на PostgreSQL без смены доменной модели.

9. Что не использовать как основной GUI

  • DB Browser for SQLite — хорош для разработчика, но не для операторского CRUD.
  • TablePlus / Beekeeper Studio — отличные dev/admin-клиенты, но не бизнес-GUI.
  • Любой GUI с прямым редактированием history-таблиц без доменных ограничений.

10. Практическая рекомендация по стеку

Рекомендуемый стартовый стек:

  • SQLite как основная БД
  • Файловое/object storage для HTML/PDF-артефактов
  • NocoDB как быстрый CRUD/backoffice GUI
  • Отдельный backend/service для операций freeze, send, sign, render, create correction

10.1 Готовые артефакты в репозитории

  • db/sqlite/work_acts_schema.sql — реальная SQLite schema.sql
  • db/sqlite/work_acts_seed_demo.sql — demo seed для красивого старта в NocoDB
  • docs/work-acts-nocodb-views-plan.html — план подключения views и режимов доступа в NocoDB

11. Если нужен совсем быстрый MVP

  1. Поднять SQLite.
  2. Создать таблицы актов, строк, ревизий, артефактов.
  3. Подключить NocoDB только к draft-CRUD и просмотру истории.
  4. Сделать 4 backend-операции: freeze revision, render html, send, sign.

Такой MVP будет быстрым, но не разрушит ключевую идею — хранить историю документов корректно.
