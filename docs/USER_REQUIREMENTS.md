# User Requirements Specification — pacus HTML Artifacts

**Version**: 1.0  
**Effective Date**: 2026-05-01  
**Standard**: ISO 9241-210:2010 Clause 6.3  

## 1. Introduction

This document specifies user requirements for HTML artifacts generated by the pacus system (project cards, work act documents, audit trails, journal entries). Requirements are derived from user needs and context of use (see `HCD_CONTEXT_OF_USE.md`).

---

## 2. Intended Context of Use

- **Users**: Project managers, accountants, auditors, system administrators
- **Tasks**: Generate, view, print work act documents and related artifacts
- **Environment**: Office setting, desktop/laptop, modern browser, localhost:4000 print server
- **Output**: Printable A4 documents (HTML → PDF → print)

---

## 3. User Requirements

### 3.1 Functional Requirements

#### FR-01: Printable Document Generation
**Description**: System shall generate HTML artifacts suitable for printing on A4 paper.

**Acceptance Criteria**:
- HTML includes `@page { size: A4 }` CSS
- Content fits within A4 margins (16mm 14mm 18mm)
- Print preview renders correctly in Chrome, Firefox, Safari

**Priority**: High  
**Verification**: Print preview test on 3 major browsers

---

#### FR-02: Project Card Display
**Description**: System shall generate project card HTML showing project details, status history, work acts, and journal entries.

**Acceptance Criteria**:
- Displays project code, name, status, dates
- Shows counterparty name and tenant
- Lists all work acts with links to HTML and audit
- Shows project status history in table format
- Includes journal entries with linked acts

**Priority**: High  
**Verification**: Visual inspection of generated HTML

---

#### FR-03: Work Act Document Access
**Description**: System shall provide links to work act HTML documents from project card and index.

**Acceptance Criteria**:
- Project card includes "HTML" link for each act
- Index page lists all acts with HTML links
- Links point to correct artifact paths

**Priority**: High  
**Verification**: Click all HTML links in generated artifacts

---

#### FR-04: Audit Trail Access
**Description**: System shall provide links to audit trail HTML for each work act.

**Acceptance Criteria**:
- Project card includes "Аудит" link for each act
- Audit trail shows status history and revisions
- Related documents section links back to project and correction acts

**Priority**: High  
**Verification**: Navigate from project card → act → audit → back

---

#### FR-05: Currency Display Flexibility
**Description**: System shall allow users to switch currency symbol (USD, EUR, RUB, GBP) via client-side JavaScript.

**Acceptance Criteria**:
- Currency selector JS present in HTML (`CURRENCY` variable)
- Switching currency updates all amount displays
- Currency code displayed alongside amounts (e.g., "USD", "RUB")

**Priority**: Medium  
**Verification**: Open HTML, change currency, verify amounts update

---

#### FR-06: Navigation Between Related Documents
**Description**: System shall provide navigation links between related documents (project ↔ acts ↔ audit ↔ journal).

**Acceptance Criteria**:
- Project card links to audit trails
- Audit trail links back to project card and work act
- Journal entry links to project card
- "К списку" (back to list) link present on all pages
- "Печать" (print) button present on all pages

**Priority**: Medium  
**Verification**: Follow all navigation links in generated artifacts

---

#### FR-07: Print Functionality
**Description**: System shall provide a print button that triggers browser print dialog.

**Acceptance Criteria**:
- Print button present in top bar (`.no-print` class)
- Clicking button calls `window.print()`
- Print button hidden in printed output (CSS `@media print`)

**Priority**: High  
**Verification**: Click print button, verify dialog appears

---

### 3.2 Usability Requirements

#### UR-01: Time to Print < 3 Clicks
**Description**: User shall be able to print any document within 3 clicks from the index page.

**Measurable Objective**:
- Click 1: Open index.html
- Click 2: Click on document link
- Click 3: Click "Печать" button

**Priority**: High  
**Verification**: Observed user test with 5 participants

---

#### UR-02: Information Scannability
**Description**: Users shall be able to scan key information (project code, act number, amounts) within 5 seconds.

**Measurable Objective**:
- Tables used for structured data
- Clear headings (h1, h2) for sections
- Amounts right-aligned and highlighted
- Status displayed prominently

**Priority**: Medium  
**Verification**: Eye-tracking study or timed user test

---

#### UR-03: Language Clarity
**Description**: All UI text shall be in clear, professional Russian.

**Measurable Objective**:
- No untranslated English terms (except technical terms like "HTML")
- Consistent terminology (e.g., always "Акт" not "Акт работ")
- Grammatically correct Russian sentences

**Priority**: High  
**Verification**: Native Russian speaker review

---

### 3.3 Accessibility Requirements

#### AR-01: Keyboard Navigation
**Description**: All links and buttons shall be accessible via keyboard (Tab, Enter).

**Acceptance Criteria**:
- All interactive elements are focusable
- Tab order follows logical reading order
- Enter/Space activates focused element

**Priority**: Medium  
**Verification**: Navigate entire artifact using only keyboard

---

#### AR-02: Print-Only Content Control
**Description**: Non-essential elements (buttons, links) shall be hidden when printing.

**Acceptance Criteria**:
- `.no-print` CSS class hides elements in `@media print`
- Printed output contains only content (no buttons, nav links)
- Page breaks avoided inside tables/rows (CSS `break-inside: avoid`)

**Priority**: High  
**Verification**: Print preview check, verify no interactive elements

---

### 3.4 Performance Requirements

#### PR-01: Artifact Generation Speed
**Description**: System shall generate HTML artifacts in < 1 second per document.

**Measurable Objective**:
- 100 project cards generated in < 100 seconds
- 500 work act audits generated in < 500 seconds

**Priority**: Low  
**Verification**: Time `gen_artifacts.py` execution for full dataset

---

#### PR-02: Page Load Speed
**Description**: Generated HTML shall load in < 2 seconds on localhost.

**Measurable Objective**:
- HTML file size < 500KB (no external resources)
- CSS embedded inline (no network requests)
- JavaScript minimal (< 50 lines)

**Priority**: Medium  
**Verification**: Browser DevTools Network tab, check load time

---

## 4. Organizational Requirements

### 4.1 Compliance
- Generated documents must include audit trail references
- Status histories must show complete transition log
- Document immutability must be preserved (no edit after generation)

### 4.2 Branding
- Documents should display tenant name
- Professional appearance expected (clean typography, consistent colors)

---

## 5. Trade-offs and Rationales

| Decision | Rationale |
|----------|----------|
| Inline CSS (not external file) | Simplifies deployment; single file per artifact |
| Client-side JS for currency | Avoids server-side regeneration; instant feedback |
| Russian language only | Target user base is Russian-speaking organizations |
| Print-focused design | Primary use case is document printing, not screen viewing |

---

| Requirement Type | Verification Method | Frequency |

| Accessibility (AR-*) | Keyboard navigation test | Every release |
