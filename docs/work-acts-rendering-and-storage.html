Рендеринг и хранение HTML для Актов работ

Документ описывает pipeline генерации печатных HTML/PDF-артефактов, хранение файлов, versioning шаблонов и требования к неизменяемости.

1. Главный принцип

Печатная форма не должна строиться на лету из текущих данных акта для уже зафиксированной или подписанной версии. Для каждой ревизии хранится отдельный неизменяемый артефакт.

2. Pipeline

  1. Пользователь формирует или меняет draft-акт.
  2. Система фиксирует work_act_revision со snapshot.
  3. Рендерер получает snapshot + template_version.
  4. На выходе формируется статический HTML.
  5. HTML сохраняется в object storage и регистрируется в document_artifact.
  6. При необходимости отдельным шагом генерируется PDF.

4. Требования к HTML

  • Формат A4, корректный print CSS.
  • Без клиентского JavaScript для вычисления содержимого.
  • Повторяемая шапка таблицы на новых страницах.
  • Контроль page-break для итогов и блоков подписи.
  • Минимум внешних зависимостей; предпочтительно встроенный CSS.
  • Без редактируемых полей, кнопок и SPA-компонентов.

6. Template versioning

  • Каждый шаблон имеет явный template_version.
  • Смена шаблона не должна изменять уже выпущенные документы.
  • Новые ревизии могут рендериться новым шаблоном, старые остаются на старом.
  • Подписанные акты не перерендериваются новым шаблоном поверх старого артефакта.

7. Политика неизменяемости

СостояниеМожно перегенерировать?Комментарий
draftДаРазрешено до фиксации отправки/подписи
generatedДа, но только в новую ревизиюСтарый артефакт остаётся в истории
sentТолько в новую ревизию до подписиНельзя терять отправленный вариант
signedНетImmutable-артефакт

8. Безопасность

  • Все пользовательские поля должны проходить escaping/санитизацию при рендеринге.
  • Не хранить артефакты в публичном доступе без авторизации.
  • Для чувствительных данных — аудит доступа к скачиванию.
  • Указывать content-type строго по типу артефакта.

9. Архивирование и retention

  • Подписанные документы хранятся по политике бухгалтерского/юридического retention.
  • Неподписанные черновые артефакты можно чистить по TTL, если они не являются частью истории.
  • Перед удалением draft-артефактов нужна проверка, что версия не закреплена в статусе sent или signed.
