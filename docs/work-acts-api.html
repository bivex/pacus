API и операции для Актов работ

Документ описывает прикладные операции, транзакционные границы, допустимые переходы статусов и контракты для хранения истории и генерации печатных артефактов.

1. Границы bounded context

  • Контекст отвечает за акты, их строки, ревизии, статусы и печатные артефакты.
  • Контекст не владеет договором или справочником контрагентов, а только ссылается на них.
  • Подписание может интегрироваться с внешним ЭДО, но внутренний статус и история хранятся локально.

3. Основные query-операции

  • GetAct — карточка акта + текущие строки.
  • ListActs — список актов по фильтрам.
  • ListActRevisions — история версий.
  • GetRevisionArtifact — HTML/PDF конкретной ревизии.
  • ListCorrections — корректировочные документы по исходному акту.

Обратные переходы signed -> draft и редактирование подписанной ревизии запрещены.

6. Транзакционные границы

  • Операции изменения статуса выполняются в коротких транзакциях.
  • Создание ревизии и запись в work_act_status_history/audit_event должны происходить атомарно.
  • Если рендеринг HTML долгий, допускается схема: сначала ревизия + job, потом артефакт и перевод в generated.
  • Нельзя держать транзакцию открытой на время сетевого вызова во внешний ЭДО.

7. Идемпотентность и конкурентный доступ

  • Для send и sign рекомендуется idempotency key.
  • Для обновления draft-акта нужен optimistic locking: версия записи или updated_at.
  • Повторный вызов SignAct для уже подписанного акта должен возвращать уже зафиксированный результат.
