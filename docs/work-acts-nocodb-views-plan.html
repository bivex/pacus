NocoDB views plan для Актов работ

Документ задаёт, какие таблицы и SQL views показывать в NocoDB, что можно редактировать, а что должно оставаться только для чтения.

1. Цель

NocoDB используется как быстрый backoffice GUI, но не как движок бизнес-логики документов. Статусы, фиксация ревизий, рендеринг HTML и подписание должны выполняться через backend/service.

3. Что не надо напрямую редактировать

  • work_act_revision — история версий, только чтение.
  • document_artifact — артефакты, только чтение.
  • work_act_status_history — append-only история.
  • audit_event — append-only аудит.
Если дать оператору прямой CRUD на эти сущности, история документов станет недостоверной.

5. Ролевая модель

РольДоступОграничения
ОператорСправочники + draft CRUD + read viewsНельзя менять signed/history/artifacts
БухгалтерRead реестр, история, артефактыБез прямой правки immutable-данных
АдминистраторШире read, ограниченный CRUD справочниковДаже админ не правит audit/history вручную

6. Какие действия оставить вне NocoDB

  1. FreezeRevision
  2. RenderRevisionHtml
  3. SendAct
  4. SignAct
  5. CreateCorrectionAct

Эти действия должны вызываться из backend/API, потому что они одновременно меняют несколько сущностей и обязаны соблюдать инварианты.

7. Практическая схема запуска

  1. Применить db/sqlite/work_acts_schema.sql.
  2. Загрузить db/sqlite/work_acts_seed_demo.sql для demo-данных.
  3. Подключить SQLite-файл к NocoDB.
  4. Показать операторам только разрешённые таблицы и views.
  5. Скрыть history/artifact raw tables из редактируемого меню.
  6. Подключить backend endpoints для статусных операций.
