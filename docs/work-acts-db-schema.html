Схема БД для Актов работ

Физическая модель данных для хранения актов, строк акта, истории версий, печатных HTML-артефактов и статусов жизненного цикла.

Если в качестве движка выбран SQLite, физические типы и миграционные правила уточняются в документе work-acts-sqlite-and-gui.html.

1. Принципы схемы

  • Текущие рабочие данные акта хранятся нормализованно.
  • Каждая зафиксированная версия хранится отдельно и не перезаписывается.
  • Печатный HTML хранится как отдельный неизменяемый артефакт версии.
  • Все связи выражаются через FK, а инварианты — через UNIQUE и CHECK.
  • Деньги хранятся в DECIMAL, время — в UTC.

Не нужно индексировать каждую колонку по отдельности; индексы должны следовать реальным запросам списка актов, карточки акта и просмотра истории.

8. Правила неизменяемости

  1. После перехода акта в signed ссылка на signed_revision_id больше не меняется.
  2. Ревизия с is_immutable = true не редактируется и не удаляется.
  3. Артефакт, связанный с immutable-ревизией, не перезаписывается.
  4. Исправления после подписи оформляются новой ревизией типа correction или новым документом, связанным с исходным актом.

9. Миграции и zero-downtime

  • Сначала добавлять новые nullable-колонки и новые таблицы.
  • Потом выполнять backfill данных по частям.
  • Только после этого переключать код на новые поля.
  • Удаление старых колонок — отдельной поздней миграцией.
  • Новые NOT NULL и тяжелые индексы вводить после заполнения данных.

10. Минимальный обязательный набор ограничений

Обязательные ограничения:
  • FK на все связи между актом, строками, ревизиями и артефактами.
  • UNIQUE (tenant_id, act_number) для номера акта.
  • UNIQUE (act_id, revision_no) для версий.
  • CHECK на допустимые статусы.
  • CHECK на неотрицательные суммы.
  • Частичный unique для одной текущей ревизии.
