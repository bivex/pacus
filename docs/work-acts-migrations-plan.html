План миграций для Актов работ

Документ задаёт порядок внедрения схемы БД и правила будущих zero-downtime изменений для истории версий и печатных артефактов.

Для SQLite нужно учитывать ограничения ALTER TABLE и чаще использовать схему: новая таблица → копирование данных → rename.

1. Цели миграционного плана

  • Создать базовую схему без ручных правок в проде.
  • Обеспечить безопасную эволюцию модели версий и артефактов.
  • Не ломать чтение старых актов при добавлении новых полей и правил.

3. Правила zero-downtime изменений

  1. Сначала добавлять новые таблицы или nullable-колонки.
  2. Потом выкатывать код, который умеет читать старый и новый формат.
  3. Затем выполнять backfill небольшими пачками.
  4. После backfill включать новые ограничения NOT NULL, CHECK, индексы.
  5. Удаление старых полей — последним этапом, после полной миграции кода.

4. Пример безопасого изменения

Пример: нужно добавить source_act_id для корректировок.
Шаг 1 — добавить nullable-колонку и индекс.
Шаг 2 — выкатить код, который умеет создавать и читать связь.
Шаг 3 — заполнить связи для исторических данных.
Шаг 4 — включить бизнес-валидации на новые сценарии.
Шаг 5 — при необходимости добавить stricter checks.

6. Валидация миграций

  • Миграции применяются на пустую БД и на БД с тестовыми данными.
  • Проверяется создание акта, фиксация ревизии, генерация HTML, перевод статусов.
  • Проверяется, что уникальные ограничения реально ловят дубли.
  • Проверяется, что подписанный акт нельзя случайно изменить.

7. Риски и меры

РискМера

Неполный backfillЗапускать верификационные запросы до включения ограничений
Несовместимость кода и схемыBackward-compatible rollout в 2–3 фазы
Повреждение истории документовОтдельные миграции для immutable-правил и обязательные интеграционные тесты

8. Наблюдаемость rollout

  • Логировать время выполнения каждой миграции.
  • Следить за lock wait, deadlock, ошибками FK/UNIQUE/CHECK.
  • После rollout проверять количество актов, ревизий и артефактов контрольными SQL-запросами.
